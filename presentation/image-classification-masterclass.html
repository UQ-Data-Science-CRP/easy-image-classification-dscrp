1 · Computer Vision

What is computer vision?

  • Teaching computers to extract meaning from pixels
  • Models learn spatial patterns inside an image (edges → textures → parts → objects)
  • Modern computer vision is dominated by deep neural networks (CNNs, Vision Transformers)

What is Roboflow

The computer vision workflow

Other aspects of computer vision

Data High quality, labelled data is most important ‘asset’ in computer vision.

Compute AI is becoming more accessible, adding more pressure into processing infrastructure (GPUs).

Models There are many pre-trained or foundation models you can already download.

You almost never need to train a model from scratch. Assume there is a pre-trained model that already does 70% of your job.

2 · Classification vs Detection vs Segmentation

The three core CV tasks

Task Output Answers Typical use
Classification One (or many) labels per image What is in this image? Species ID, quality control
Object detection Labels + bounding boxes What and where? Counting, tracking, surveillance
Segmentation Per-pixel labels (mask) What, where, exact shape? Medical imaging, area/biomass

Warning

Cost scales with specificity. Segmentation masks take ~10× longer to annotate than classification labels. Pick the minimum task that answers your question.

3 · Training, Evaluation & Testing

The three-split doctrine

Important

The test set is sacred. If you look at it more than once, you have leaked information and your metric is optimistic.

This is the single most violated rule in applied ML. Researchers re-touch their test set dozens of times. Recommend lockbox protocol: commit test set to a separate file and don’t open until final evaluation. Mention nested cross-validation for small data.

Splitting matters — stratify & group

from sklearn.model_selection import train_test_split, GroupShuffleSplit

# Stratify by label so class ratios are preserved
X_tv, X_test, y_tv, y_test = train_test_split(
    X, y, test_size=0.15, stratify=y, random_state=42
)

# If you have groups (same site, same camera, same patient) — split by GROUP
gss = GroupShuffleSplit(n_splits=1, test_size=0.15, random_state=42)
train_idx, test_idx = next(gss.split(X, y, groups=site_id))

Warning

If frames from the same video/site/patient end up on both sides of your split, your metric is fiction.

Classic failure mode in ecological imagery: frames 1-100 of a transect land in train, frames 101-200 in test — near-duplicates → 99% accuracy → deploys → 60% accuracy. Group-aware splitting is mandatory for most field data.

A minimal training loop

import torch, torch.nn as nn

model, opt = build_model().cuda(), torch.optim.AdamW(model.parameters(), 1e-4)
loss_fn = nn.CrossEntropyLoss()

for epoch in range(EPOCHS):
    model.train()
    for xb, yb in train_loader:
        opt.zero_grad()
        loss = loss_fn(model(xb.cuda()), yb.cuda())
        loss.backward(); opt.step()

    model.eval()
    with torch.no_grad():
        val_acc = evaluate(model, val_loader)
    print(f"epoch {epoch}: val_acc={val_acc:.3f}")
    save_if_best(model, val_acc)           # early stopping

Underline: save best on validation, NOT last epoch. Talk about learning rate schedules, early stopping patience, and the fact that most practical training is now just “fine-tune for 10-30 epochs, pick best validation checkpoint.”

Reading training curves

Healthy: train ↓ and val ↓ together, then val plateaus.

Overfitting: train ↓, val ↑ (or flat). Add regularisation, augment, get more data.

Underfitting: both curves high, flat. Bigger model, train longer, check data.

Broken: loss explodes / NaN. Lower LR, check labels, check preprocessing.

Push: always plot curves. Metrics-only workflows hide problems. TensorBoard, wandb, or even matplotlib — doesn’t matter, just plot.

4 · Domain Shift

What is domain shift?

Domain shift = when the distribution of data at deployment differs from the distribution at training.

Common flavours:

  • Covariate shift — input pixels differ (new camera, lighting, season)
  • Label shift — class frequencies change (rare species becomes common)
  • Concept drift — the meaning of the label changes over time (new disease morphology)

Warning

Domain shift is the #1 reason research models fail in the real world. Accuracy on a held-out test set predicts almost nothing about accuracy in a new environment.

This is where “it worked in the lab” projects go to die. Examples from your domain: camera swap between field campaigns, turbidity change in water, new biofouling pattern on cages. Encourage: always evaluate on a held-out deployment domain, not just random split.

Examples & quick detection

Examples - Model trained on iPhone photos → deployed on drone - Daytime training → dawn/dusk deployment - Great Barrier Reef corals → Red Sea corals - Healthy crops 2024 → drought crops 2026

How to detect it - Compare feature embeddings (e.g. CLIP features) between train and deploy sets - Monitor prediction confidence distribution over time - Track class ratios in production - Periodic spot-checking by a human expert

# Quick "is my new data different?" check using embeddings
from sentence_transformers import SentenceTransformer
from scipy.spatial.distance import cdist

enc = SentenceTransformer("clip-ViT-B-32")
train_emb = enc.encode(train_images)
new_emb   = enc.encode(new_images)
mean_dist = cdist(train_emb.mean(0, keepdims=True), new_emb).mean()

Frame: “your metric is only as good as the assumption that tomorrow looks like yesterday.” Recommend building a small drift-monitoring dashboard into any real deployment.

Mitigations

  • Collect a little data in the new domain and fine-tune (usually best ROI)
  • Aggressive augmentation that simulates the shift (lighting, blur, colour jitter)
  • Domain adaptation techniques (DANN, CORAL) — rarely worth the complexity in practice
  • Foundation models — large pre-trained features are more robust out-of-the-box
  • Human-in-the-loop for the first N predictions in a new domain

Tip

A 50-image fine-tune in the new domain usually beats a cleverer domain-adaptation algorithm on the old data.

Be honest: domain adaptation is an active research area but in applied settings the pragmatic answer is almost always “label a few hundred images in the new domain and fine-tune.” Tie this forward to transfer learning.

5 · Transfer Learning

Why it works

  • Early layers learn generic features (edges, textures)
  • Late layers learn task-specific features
  • Swap the final layer → keep the rest
  • Small dataset? Only fine-tune the head. Bigger? Unfreeze more.

Transfer learning turns a 100k-image problem into a 1k-image problem.

Rule of thumb: with <1k images, only train the head. 1k-10k: unfreeze last block. >10k: unfreeze everything but keep a low LR on the backbone. Mention discriminative LRs (smaller LR for earlier layers).

Transfer learning in 15 lines

import torch, torch.nn as nn
from torchvision.models import resnet50, ResNet50_Weights

model = resnet50(weights=ResNet50_Weights.DEFAULT)

# 1) freeze the backbone
for p in model.parameters():
    p.requires_grad = False

# 2) replace the classification head (ImageNet had 1000 classes)
NUM_CLASSES = 5
model.fc = nn.Linear(model.fc.in_features, NUM_CLASSES)

# 3) only the new head is trainable
opt = torch.optim.AdamW(model.fc.parameters(), lr=1e-3)

# 4) train as usual (see Section 3)

Point out: timm library has hundreds of backbones with the same API. Newer recipes often just use a Vision Transformer (ViT) backbone — convnets still competitive for small data.

6 · Foundational Models

What are they?

Models trained on internet-scale data that can be used zero-shot or with minimal adaptation for many downstream tasks.

Model What it does Use it for
CLIP Aligns image ↔︎ text embeddings Zero-shot classification, retrieval, drift detection
DINOv2 Self-supervised image features Strong backbone, embeddings, clustering
SAM / SAM 2 Promptable segmentation Annotation acceleration, masks without training
GroundingDINO Open-vocabulary detection “Find anything” from a text prompt
YOLO-World Real-time open-vocab detection Fast detection without retraining

Foundational models have radically changed the workflow. 3 years ago you’d spend 2 weeks training a classifier from scratch. Now: CLIP zero-shot → SAM masks → GroundingDINO boxes → fine-tune only if needed. Mention licence gotchas (some are non-commercial).

Zero-shot classification with CLIP

import torch, clip
from PIL import Image

device = "cuda" if torch.cuda.is_available() else "cpu"
model, preprocess = clip.load("ViT-B/32", device=device)

image = preprocess(Image.open("coral.jpg")).unsqueeze(0).to(device)
texts = clip.tokenize([
    "a photo of healthy coral",
    "a photo of bleached coral",
    "a photo of dead coral",
]).to(device)

with torch.no_grad():
    logits_per_image, _ = model(image, texts)
    probs = logits_per_image.softmax(dim=-1).cpu().numpy()

print(probs)   # e.g. [[0.78, 0.18, 0.04]]

Tip

Use CLIP zero-shot as your baseline. If a fine-tuned model can’t beat it, something is wrong.

CLIP as baseline is a habit that saves projects. Also mention: CLIP features are excellent inputs to a simple logistic regression — often matches a from-scratch CNN with 1% of the effort.

SAM: masks without training

from segment_anything import SamAutomaticMaskGenerator, sam_model_registry

sam = sam_model_registry["vit_h"](checkpoint="sam_vit_h.pth").to("cuda")
masks = SamAutomaticMaskGenerator(sam).generate(image_rgb)
# masks is a list of dicts: {'segmentation': ndarray, 'area': int, ...}

Note

SAM gives you masks but not labels. Pair it with CLIP or a small classifier for “mask → label”.

The “SAM + CLIP” combo is the modern annotation accelerator: one click per object, automatic labels, human review. Roboflow bakes this in as “Smart Polygon” and similar tools.

7 · Evaluation — Beyond F1

F1 is a starting point, not the finish line

Warning

A single F1 number hides where and why your model fails. Stakeholders deserve — and need — more.

Questions a single F1 can’t answer:

  • Does it fail on the rare class or the common class?
  • Is it over-confident when wrong?
  • How does it degrade across subgroups (sites, seasons, camera types)?
  • Are errors safety-critical or merely annoying?
  • Is there a threshold I should choose differently?

Reframe: F1 is a compressed summary — useful for model selection, terrible for deployment decisions. Push the audience to always bring at least 3 views to a stakeholder: confusion matrix, per-class metrics, calibration.

A richer evaluation toolkit

Tool What it tells you When to use
Confusion matrix Per-class errors & confusions Always
Per-class precision/recall Class imbalance effects Imbalanced data
ROC & PR curves Threshold trade-offs Binary / one-vs-rest
Calibration plot Are confidences trustworthy? Any probabilistic output
Cohen’s κ / MCC Agreement beyond chance Imbalanced or ordinal
mAP / IoU Detection & segmentation quality Detection / segmentation
Subgroup analysis Fairness & drift hotspots Always, if metadata exists
Error case gallery Qualitative failure modes Before deployment

Emphasise the last two: subgroup analysis and error-case galleries have saved more projects than fancy metrics. Build an “error wall” — literally a folder of the 50 worst predictions — and review it with domain experts.

Code: the evaluation essentials

from sklearn.metrics import (classification_report, confusion_matrix,
                             roc_auc_score, cohen_kappa_score)
from sklearn.calibration import calibration_curve
import matplotlib.pyplot as plt

print(classification_report(y_true, y_pred, digits=3))
print("kappa :", cohen_kappa_score(y_true, y_pred))
print("AUROC :", roc_auc_score(y_true, y_prob, multi_class="ovr"))

# Calibration plot — are the probabilities trustworthy?
frac_pos, mean_pred = calibration_curve(y_true == 1, y_prob[:, 1], n_bins=10)
plt.plot(mean_pred, frac_pos, "o-"); plt.plot([0,1],[0,1], "--")
plt.xlabel("Predicted probability"); plt.ylabel("Observed frequency")

Tip

A well-calibrated 0.7 should happen ~70% of the time. If it happens 30% or 95% of the time, your confidences are lying.

Calibration is under-taught. For any system that triages or ranks, miscalibration is worse than miscoverage. Mention temperature scaling as a cheap post-hoc fix.

For detection & segmentation

  • IoU (Intersection over Union) — overlap between prediction and truth
  • mAP @ 0.5 — mean Average Precision at IoU threshold 0.5 (lenient)
  • mAP @ 0.5:0.95 — averaged over 10 IoU thresholds (COCO standard, strict)
  • Per-class AP — same story as per-class F1
  • Boundary F-score — specifically for segmentation edge quality

Note

mAP can hide systematic issues. Always visualise predictions on ~50 images before trusting a number.

Stress visual inspection. I have seen mAP 0.85 models that were unusable because they missed the one species that actually mattered to the ecologist.

8 · Deployment

What “deployment” actually means

Shipping weights is 20% of deployment. The other 80% is monitoring, retraining, and the humans in the loop.

This is the biggest gap between academic ML and production ML. Most research models never cross this line. The slide after next is the checklist you should answer before shipping.

The deployment checklist

Performance

  • Target latency (ms/frame)?
  • Throughput (images/sec)?
  • Batch vs streaming?

Hardware

  • CPU / GPU / edge (Jetson, Coral)?
  • Offline vs connected?
  • Memory ceiling?

Data pipeline

  • Pre/post-processing matches training?
  • Input validation & logging?
  • Fallback when input is bad?

Operations

  • Versioning of model + data + code?
  • Rollback plan?
  • Monitoring (drift, latency, errors)?
  • Retraining cadence?

Humans & ethics

  • Who reviews flagged cases?
  • Who is accountable for errors?
  • Privacy / consent of imagery?
  • Bias & fairness audits?

Cost

  • $ per 1000 inferences?
  • $ per retrain cycle?

Walk through each bullet at pace. Highlight: “pre/post-processing matches training” is the silent killer — a 0.5 drop in mean normalisation can halve accuracy. Also: privacy of human subjects in imagery is increasingly non-optional.

Serving patterns

Pattern When Example
Batch You have a pile of images to process overnight Cron + script on a GPU node
REST API Web/mobile apps need on-demand inference FastAPI + ONNX on a cloud GPU
Edge Network/privacy/latency constraints Jetson Nano / Coral TPU
Managed You don’t want to babysit infrastructure Roboflow Hosted, Replicate, Modal 
# Minimal FastAPI inference service
from fastapi import FastAPI, UploadFile
from PIL import Image
import io, torch

app = FastAPI(); model = torch.jit.load("model.ts").eval()

@app.post("/predict")
async def predict(file: UploadFile):
    img = preprocess(Image.open(io.BytesIO(await file.read())))
    with torch.no_grad():
        probs = model(img.unsqueeze(0)).softmax(-1).tolist()[0]
    return {"probs": probs}

For 90% of research projects, “managed” is the right answer. The time cost of self-hosting GPUs rarely pays back. Roboflow’s hosted inference is a concrete option we’ll see in the practical.

9 · AI vs Expertise

What you can confidently do yourself

DIY-friendly

  • Zero-shot classification with CLIP on clear categories
  • Fine-tune a pre-trained classifier on 500-5000 labelled images
  • Run inference with pre-trained detection/segmentation models
  • Use Roboflow / Ultralytics / HuggingFace end-to-end
  • Simple evaluation dashboards (confusion matrix + curves)

Call-an-expert

  • Deployment at scale (1000s of req/sec, edge fleets)
  • Safety-critical decisions (medical, autonomous, regulatory)
  • Novel modalities (multi-spectral, 3D, video, time-series)
  • Custom architectures or loss functions
  • Dataset strategy for very rare classes / severe imbalance
  • MLOps, monitoring, drift detection pipelines

Reframe the question from “can AI do this?” to “can I do this with today’s tools, and is the risk of being wrong acceptable?” Emphasise: the tools are genuinely great for 80% of problems. The remaining 20% eat 80% of the effort — that’s where specialists earn their keep.

11 · Ensembles & Workflows

Ensembles: combining multiple models

Why ensemble?

  • Different models make different mistakes
  • Averaging reduces variance
  • Typical gain: +1-5% accuracy, “for free”

Types

  • Averaging — average softmax outputs
  • Voting — majority class wins
  • Stacking — a meta-model learns how to combine
  • Test-time augmentation (TTA) — same model, multiple augmented views

Pragmatic note: in industry, ensembles are often avoided because of latency. In research, use them shamelessly — they’re a free accuracy boost. TTA is the cheapest ensemble of all (no extra training).

Code: averaging & TTA

# --- Model averaging ---
import torch.nn.functional as F

def ensemble(models, x):
    probs = [F.softmax(m(x), dim=-1) for m in models]
    return torch.stack(probs).mean(0)

# --- Test-time augmentation ---
def tta_predict(model, x, augs):
    preds = [F.softmax(model(aug(x)), dim=-1) for aug in augs]
    return torch.stack(preds).mean(0)

augs = [
    lambda x: x,
    lambda x: torch.flip(x, dims=[-1]),   # horizontal flip
    lambda x: torch.flip(x, dims=[-2]),   # vertical flip
]

TTA typically gives +0.5-2% with no retraining. Warn: keep the same augmentations at eval that make sense for the domain — flipping microscopy might be invalid.

Workflows: cascades & humans-in-the-loop

Ensembles combine models. Workflows combine steps. In production this often matters more.

A clever workflow often beats a smarter model. Think in terms of pipelines, not monoliths.

Real-world tip: a fast “reject” stage (is there even an object here?) + a slow accurate stage on the hits is cheaper and better than one big model on every frame. Also: a confidence-thresholded human-review stage is the single highest-leverage workflow pattern.

A small real-world pattern

# Two-stage cascade with human-in-the-loop
from ultralytics import YOLO
detector  = YOLO("yolov8n.pt")
classifier = torch.jit.load("fine_grained_classifier.ts").eval()

def pipeline(image, conf_threshold=0.85):
    detections = detector(image)[0].boxes
    outputs = []
    for box in detections:
        crop = crop_from_box(image, box)
        probs = classifier(preprocess(crop).unsqueeze(0)).softmax(-1)
        conf, cls = probs.max(-1)
        if conf.item() < conf_threshold:
            outputs.append({"bbox": box, "label": None, "needs_review": True})
        else:
            outputs.append({"bbox": box, "label": cls.item(), "conf": conf.item()})
    return outputs

Walk through: detect → crop → classify fine-grained → threshold for human review. This single pattern covers a huge fraction of applied CV projects. Roboflow Workflows (their new feature) encodes exactly this.
